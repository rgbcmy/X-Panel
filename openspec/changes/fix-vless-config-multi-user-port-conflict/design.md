## 背景

V2Board 同步任务（`web/job/v2board_sync_job.go`）负责为 V2Board UniProxy 用户 API 返回的用户配置 VLESS 入站。每个用户可能携带一个包含 `inbound_port` 和 `inbound_tag` 的 `vless_config` 块。当前逻辑只通过 `inbound_tag` 查找已有入站；若未找到则调用 `AddInbound`，该方法内部强制执行端口唯一性校验。当两个或更多用户共享相同的 `inbound_port` 时，第一个用户之后的所有用户都会触发"端口已存在"错误，并被静默计入错误统计，导致除第一个用户外的所有用户均未被配置。

## 目标 / 非目标

**目标：**
- 确保每个带有 `vless_config` 的用户均能被配置为正确入站的客户端，即使多个用户共享相同的 `inbound_port`。
- 将修复范围最小化，仅限于 `processUserInbound`——不涉及数据结构变更，不涉及 API 变更。

**非目标：**
- 修改入站的命名、标签或排序方式。
- 变更流量上报、xray 重启逻辑或其他同步相关逻辑。
- 在本次变更中支持 VLESS 以外的协议。

## 技术决策

### 决策：创建前先进行端口回退查找

**方案**：在 `processUserInbound` 中，当基于标签的查找未找到结果后，再通过端口进行一次查找（`getInboundByPort`）。若找到端口匹配的入站，则对其调用 `updateUserInbound`，而不是新建入站。

**理由**：根本原因是缺少回退逻辑——此修复沿用了标签查找中已有的"查找或创建"模式，将其延伸一层。无需变更数据模型。

**已考虑的替代方案**：
- *提前将所有用户合并到一个入站*：需要重新设计同步循环，可能破坏按用户分标签的语义，改动过于侵入性。
- *捕获"端口已存在"错误后重试*：错误驱动的流程较为脆弱；主动查找更清晰，也避免了不必要的创建尝试。

### 决策：新增辅助方法 `getInboundByPort`

引入一个专用辅助方法，遍历 `GetAllInbounds()` 并返回第一个 `Port` 与请求端口匹配的入站（暂时忽略 `Listen` 地址，因为所有受管入站均使用 `""`）。未匹配时返回 `nil, error("record not found")`——与 `getInboundByTag` 保持一致。

## 风险 / 权衡

- **同一端口存在多个入站（边缘情况）** → `getInboundByPort` 返回第一个匹配项。若两个入站意外共享同一端口（`AddInbound` 自身的守卫机制不允许此情况），用户将被加入第一个返回的入站。缓解措施：这是 `AddInbound` 维护的现有不变量，不引入新风险。
- **性能** → 每个用户会调用两次 `GetAllInbounds()`（一次按标签，一次按端口），用户量较大时会增加数据库读取。缓解措施：在预期规模下可接受；未来可优化为每次同步周期缓存一次入站列表。

## 迁移计划

- 部署更新后的二进制文件；无需数据库迁移。
- 下一个同步周期中，此前被跳过的用户将自动作为客户端加入已有的共享入站。
- 回滚方式：回退二进制文件；已添加的客户端保留在数据库中，但后续同步将重新只处理第一个用户。
